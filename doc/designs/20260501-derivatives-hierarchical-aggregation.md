# Derivatives Hierarchical Aggregation: Gap Analysis & Migration Plan

**Date**: 2026-05-01
**Status**: Planning
**Authors**: yoh, Claude

## Summary

The hierarchical extraction pipeline for derivatives is partially implemented but not
connected end-to-end. The `bids_studies` package already has the extraction and
aggregation code (`extract_derivative_stats` → `derivatives+subjects.tsv` →
`derivatives+datasets.tsv`), and the Snakemake workflow already calls it. However,
`studies+derivatives.tsv` generation ignores these intermediate files and instead uses
a separate code path in `openneuro_studies.metadata.derivative_extractor` that
re-extracts directly from git trees.

This document clarifies the architecture, identifies the gaps, and proposes a migration.

## Current State

### What Works (Sourcedata — Full Hierarchy)

```
bids_studies.extraction.subject.extract_subjects_stats()
    → study-*/sourcedata/sourcedata+subjects[+sessions].tsv
bids_studies.extraction.dataset.aggregate_to_dataset()
    → study-*/sourcedata/sourcedata.tsv
bids_studies.extraction.study.aggregate_to_study()
    → columns in studies.tsv
```

All three levels live in `bids_studies`. The Snakemake workflow calls
`extract_study_stats()` which orchestrates the full pipeline and writes intermediate
TSVs. `studies.tsv` is then generated by `openneuro_studies` reading from these.

### What's Broken (Derivatives — Disconnected)

**Path A (bids_studies — hierarchical, already called by Snakemake):**
```
bids_studies.extraction.derivative.extract_derivative_subjects_stats()
    → study-*/derivatives/{tool}/derivatives+subjects[+sessions].tsv
bids_studies.extraction.derivative.aggregate_derivative_to_dataset()
    → study-*/derivatives/{tool}/derivatives+datasets.tsv
```

The Snakemake workflow (`code/workflow/Snakefile:196-215`) already calls
`extract_derivative_stats()` for each derivative, producing these files.

**Path B (openneuro_studies — flat, used by studies+derivatives.tsv):**
```
openneuro_studies.metadata.derivative_extractor.extract_derivative_metadata()
    → directly populates rows in studies+derivatives.tsv
openneuro_studies.metadata.studies_plus_derivatives_tsv.generate_studies_derivatives_tsv()
    → writes studies+derivatives.tsv
```

This path does NOT read from `derivatives+subjects.tsv` or `derivatives+datasets.tsv`.
It re-extracts from git trees directly, producing different (and less detailed) metrics.

### The Two `extract_derivative_stats` Functions

There are **two different functions** with the same name:
1. `bids_studies.extraction.study.extract_derivative_stats()` — hierarchical,
   uses `SparseDataset`, writes per-subject TSVs
2. `openneuro_studies.metadata.derivative_extractor.extract_derivative_stats()` —
   flat git-based extraction (size_total, file_count via git commands)

The Snakemake workflow imports from `bids_studies`. The CLI `metadata generate
--derivatives-tsv` command imports from `bids_studies` for per-derivative extraction
but from `openneuro_studies` for the top-level aggregation.

## Architecture Violations

1. **`bids_studies/extraction/study.py` imports from `openneuro_studies`**:
   ```python
   from openneuro_studies.lib.error_tracking import ErrorLevel, log_error
   ```
   This creates a circular dependency. `bids_studies` is supposed to be generic/reusable.

2. **`bids_studies/extraction/derivative.py` imports `NetworkError` from
   `openneuro_studies`** (with fallback):
   ```python
   from openneuro_studies.lib.exceptions import NetworkError
   ```

3. **`openneuro_studies.metadata.derivative_extractor`** (1026 lines) contains
   extraction logic that belongs in `bids_studies`:
   - Git tree traversal for file listing
   - Size computation from git-annex keys
   - Subject/modality counting
   - Version tracking

4. **`studies+derivatives.tsv` generation** doesn't use the hierarchical files already
   computed by the Snakemake workflow, leading to:
   - Redundant computation
   - Inconsistent metrics between per-derivative TSVs and top-level TSV
   - Temporary subdataset install/drop cycles that are unnecessary if intermediate
     TSVs already exist

## Proposed Architecture

### Principle: bids_studies Does All Extraction/Aggregation

```
bids_studies (generic library, no openneuro_studies imports):
├── extraction/
│   ├── subject.py          — per-subject sourcedata stats
│   ├── dataset.py          — per-dataset aggregation
│   ├── study.py            — per-study aggregation, file writing
│   ├── derivative.py       — per-subject derivative stats
│   └── tsv.py              — column definitions, writers
├── sparse/                 — sparse data access
└── schemas/                — JSON sidecars

openneuro_studies (application layer, uses bids_studies):
├── cli/                    — user commands
├── metadata/
│   ├── studies_tsv.py      — reads sourcedata.tsv → studies.tsv
│   ├── studies_plus_derivatives_tsv.py  — reads derivatives+datasets.tsv → studies+derivatives.tsv
│   └── ...
└── ...
```

### Target Data Flow for Derivatives

```
Snakemake extract rule (already does step 1-2):
  1. bids_studies.extraction.extract_derivative_stats()
     → study-*/derivatives/{tool}/derivatives+subjects[+sessions].tsv
     → study-*/derivatives/{tool}/derivatives+datasets.tsv
  2. Aggregate per-study: study-*/derivatives/derivatives.tsv  (NEW - one row per derivative)

studies+derivatives.tsv generation:
  3. Read study-*/derivatives/derivatives.tsv for each study
  4. Combine into top-level studies+derivatives.tsv (with study_id prefix)
```

### What Needs to Change

| Change | Package | Effort |
|--------|---------|--------|
| Remove `openneuro_studies` imports from `bids_studies/extraction/study.py` | bids_studies | Small |
| Remove `openneuro_studies` imports from `bids_studies/extraction/derivative.py` | bids_studies | Small |
| Add error logging abstraction to `bids_studies` (callback or simple file logger) | bids_studies | Medium |
| Add `derivatives.tsv` aggregation (one row per derivative within a study) | bids_studies | Medium |
| Move git-based derivative metadata extraction to `bids_studies` or keep as complementary | Both | Decision needed |
| Rewrite `studies+derivatives.tsv` generation to read from `derivatives.tsv`/`derivatives+datasets.tsv` | openneuro_studies | Medium |
| Update Makefile freshness check (currently checks sourcedata timestamps for derivatives) | Makefile | Small |

## Open Questions

1. **What metadata in `studies+derivatives.tsv` is NOT available from `bids_studies` extraction?**
   - `processed_raw_version` / `current_raw_version` / `uptodate` / `outdatedness` — version tracking
   - `tasks_processed` / `tasks_missing` / `processing_complete` — completeness analysis
   - `template_spaces` / `transform_spaces` — space detection
   - These require comparing derivative contents against the source dataset. Should this logic
     move to `bids_studies` or remain in `openneuro_studies`?

2. **Should `derivatives.tsv` (per-study aggregation) be a new file?**
   - Currently `sourcedata.tsv` is the per-study aggregation for sourcedata
   - The equivalent for derivatives would be `derivatives.tsv` (one row per derivative)
   - Or should it be `derivatives+datasets.tsv` at the study level?

3. **How to handle the git-only extraction path?**
   - `bids_studies` uses `SparseDataset` (needs cloned repo or fuse mount)
   - `openneuro_studies/derivative_extractor.py` uses bare git commands (no clone needed)
   - Both approaches have value; should `bids_studies` support git-only mode?

## Immediate Next Steps

1. Fix circular imports: remove `openneuro_studies` imports from `bids_studies`
2. Verify `derivatives+subjects.tsv` / `derivatives+datasets.tsv` are actually generated
   by current Snakemake runs (check a study directory)
3. Decide on question #1 above (what stays in openneuro_studies)
4. Implement the aggregation path from intermediate TSVs → `studies+derivatives.tsv`
